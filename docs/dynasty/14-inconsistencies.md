# Dynasty Design — Inconsistencies and Conflicts

Generated by cross-referencing all 14 design documents. Issues are grouped by severity.

---

## High Severity

These should be resolved before further design work proceeds.

---

### H1 — Coaching offers: prior-team slot vs. separate option

**Files:** `2-season-structure.md` §3, `9-off-season.md` Chapter 0

`2-season-structure.md` says returning to the prior team is *one of* the 2–3 incoming offers. `9-off-season.md` Chapter 0 lists re-signing with the current team as a distinct option *in addition to* the 2–3 offers — implying up to 4 total choices. These two descriptions are mutually contradictory.

**Resolution needed:** Decide whether the prior team counts as one slot within the 2–3, or is an extra option outside the count. Update one document to match the other.

---

### H2 — Heat state model: 3 states vs. 5 states

**Files:** `3-game-loop.md` §3

The Heat states and thresholds table defines **three** states: Hot, Neutral, Cold. The morale delta table in the same section uses **five** states: Hot (+2), Warm (+1), Neutral (0), Cool (-1), Cold (-2). "Warm" and "Cool" are used in the morale table but never defined in the Heat state definition. Either the morale table was designed for a five-state system while the rest of the document describes a three-state system, or Warm/Cool are aliases that need to be formally defined.

**Resolution needed:** Either extend the Heat state definition to five states with explicit thresholds for Warm and Cool, or collapse the morale delta table to three entries matching the defined states.

---

### H3 — `veteran-leader` morale floor of 10 is in the Frustrated range, not Stable

**Files:** `3-game-loop.md` §3, `11-role-assignment.md` §5, `13-defensive-schemes.md` §6

All three files state: `` `veteran-leader` — morale floor of 10; "never drops below Stable." ``

The morale threshold table defines Stable as 12–16 and Frustrated as 7–11. A floor of 10 is squarely in Frustrated territory, not Stable. A `veteran-leader` player at morale 10 would simultaneously be described as "protected from dropping below Stable" and be in a state that triggers narrative event risk and minutes expectation pressure.

**Resolution needed:** Either raise the floor to 12 (the actual bottom of Stable), or change the prose description to "never drops below Frustrated." Fix in all three files.

### H4 — Tag name mismatch: `shot-clock-savvy` vs. `late-clock-savvy`

**Files:** `3-game-loop.md` §1.2, `6-player-attributes-and-tags.md` §2.2

`3-game-loop.md` §1.2 (shot clock exhaustion branch) uses the tag `shot-clock-savvy`. `6-player-attributes-and-tags.md` §2.2 names the same concept `late-clock-savvy`. These are two different names for the same tag.

**Resolution needed:** Pick one name and apply it consistently across both files.

---

### H5 — `ball-stopper` tag is undefined

**Files:** `3-game-loop.md` §1.2, `6-player-attributes-and-tags.md`

`3-game-loop.md` §1.2 references the tag `ball-stopper` in the shot clock exhaustion branch ("low IQ / `ball-stopper` tag → shot clock violation"). This tag does not appear anywhere in `6-player-attributes-and-tags.md`'s IQ, Specialisation, Behavioral, or Durability tag tables.

**Resolution needed:** Either add `ball-stopper` to the tag vocabulary in `6-player-attributes-and-tags.md` with a full definition, or replace it with an existing tag.

---

### H6 — `scouting_knowledge` is a simulation input with no data model

**Files:** `3-game-loop.md` §1.2, `4-between-games.md` §3, `7-draft-and-scouting.md` §2.1

The defensive scoring function requires `scouting_knowledge(primary_defender, ball_handler)` as a live per-action numeric input. The scouting system as described in `7-draft-and-scouting.md` and `4-between-games.md` is a **progressive card reveal** — it uncovers attributes and tags on a dossier card. It does not produce a continuous numeric score. There is no defined conversion layer between "cards revealed" and the simulation parameter.

**Resolution needed:** Either define `scouting_knowledge` as a tracked numeric field (how it accumulates, its range, how between-game scouting actions increment it), or respecify the scoring function to use revealed-card flags instead.

---

### H7 — Reputation system is undefined

**Files:** `0-design-notes.md` §2, `2-season-structure.md`, `3-game-loop.md` §3, `9-off-season.md` §1.1

Reputation is the core meta-progression driver referenced across nearly every document. No file defines its numeric range, scoring formula, or the tier thresholds for New / Established / Franchise Coach. `0-design-notes.md` §2 explicitly flags this as undesigned. All tier-gated mechanics in `9-off-season.md` and `7-draft-and-scouting.md` depend on thresholds that do not exist.

**Resolution needed:** Create a dedicated section or document that specifies the reputation scoring model, numeric range, and tier boundaries.

---

### H8 — Offensive schemes have no defining document

**Files:** `3-game-loop.md` §5, `11-role-assignment.md` §4, `13-defensive-schemes.md`

`13-defensive-schemes.md` fully defines the defensive scheme vocabulary, effects, and auto-assignment logic. No equivalent exists for offensive schemes. The possession generator's `scheme_weight(play_type)` and the role assignment function both depend on offensive scheme definitions that are nowhere specified. Scheme names in `3-game-loop.md` ("push pace", "hero ball", "zone press") are informal examples, not a canonical vocabulary.

**Resolution needed:** Create an offensive schemes document equivalent to `13-defensive-schemes.md`, defining the canonical scheme names, their `scheme_weight` tables, and which role tags they reward.

---

### H9 — Injury system is a stub

**Files:** `3-game-loop.md` §6–7, `4-between-games.md` §1–2, `6-player-attributes-and-tags.md` §2.4

Multiple files reference injuries (crisis triggers, fatigue flags, emergency signings, `injury-prone` / `iron-man` / `high-motor` tags interacting with "injury risk threshold") but no document defines injury severity tiers, how many games a player misses, or how emergency signings interact with the roster and free-agent pool.

**Resolution needed:** Define a minimum viable injury system: severity levels, games-missed model, and the emergency signing flow.

---

### H10 — Player–player synergy has no hook in the possession scoring function

**Files:** `10-relationships.md` §4, `3-game-loop.md` §1.2

`10-relationships.md` §4 defines synergy bonuses as "probability bonuses on play types matching the synergy tag." The possession generator scoring function in `3-game-loop.md` includes no synergy term:

```
option_score(play_type) =
  scheme_weight(play_type)
  + play_type_recency_score(...)
  + player_fit(primary_candidate, play_type)
  + heat_modifier(primary_candidate)
  - fatigue_penalty(primary_candidate)
  + matchup_exploit(primary_candidate, defender)
```

The system has defined effects but no integration point in the simulation model.

**Resolution needed:** Either add a `synergy_modifier` term to the scoring function, or explicitly state that synergy integration is deferred and mark it as a known gap in `3-game-loop.md`.

---

### H11 — Between-game slot economy vs. off-season chapter economy

**Files:** `4-between-games.md` §1–2, `7-draft-and-scouting.md` §2.2, `9-off-season.md` Chapter 3

In-season: one free decision slot per game. Off-season scouting in `9-off-season.md` Chapter 3 grants 1 (New Coach), 2–3 (Established), or 4+ (Franchise) scouting actions per chapter step. These are different cadences and their relationship is never defined. `7-draft-and-scouting.md` §2.2 states scouting "consumes the context-gated free decision slot," but the between-game slot is a per-game resource while the off-season operates by chapter advancement. How many scouting actions a coach actually has during the pre-draft period is unresolvable.

**Resolution needed:** Clarify whether the "one free decision slot" economy applies during the off-season, or whether the chapter-action counts in `9-off-season.md` Chapter 3 replace it. Define the transition between the two economies.

## Medium Severity

### M1 — Rival displacement rule: "weakest" vs. "oldest or least-intense"

**Files:** `0-design-notes.md` §3, `8-league-simulation.md` §3.3

`0-design-notes.md` §3 says new rivalries displace "the weakest existing rivalry." `8-league-simulation.md` §3.3 says the new rivalry displaces "the oldest or least-intense active rivalry." "Oldest" and "least-intense" are different criteria and can produce different outcomes: an old, still-intense rivalry survives under `0-design-notes.md`'s rule but could be dropped under `8-league-simulation.md`'s rule.

**Resolution needed:** Settle on a single displacement criterion and update both files.

---

### M2 — "Archetype" used for both players and teams without a unified vocabulary

**Files:** `5-card-systems.md` §1 and §3, `6-player-attributes-and-tags.md` §1.2, `8-league-simulation.md` §2.1, `11-role-assignment.md` §4

Team archetypes are formally defined in `8-league-simulation.md` §2.1 with a named vocabulary. Player archetypes are referenced throughout but the vocabulary (e.g. `positionless-forward`, `3d-specialist` appear only as examples in `5-card-systems.md` §3) is never formally defined. Multiple systems depend on player archetype as an input.

**Resolution needed:** Define the player archetype vocabulary in `6-player-attributes-and-tags.md`, including how archetypes are derived from attributes and what the complete set of named archetypes is.

---

### M3 — Domain summary score derivation is undefined, but archetype depends on it

**Files:** `6-player-attributes-and-tags.md` §1.2, `5-card-systems.md` §3, `0-design-notes.md` §3

`6-player-attributes-and-tags.md` §1.2 says domain summary scores are "derived from attribute averages within each domain," but this is flagged as deferred in `0-design-notes.md` §3. The Scoring domain has 5 attributes and the Defense domain has 4; a straight average produces differently sized samples. `5-card-systems.md` §3 implies archetype is derived from something more nuanced than averages. Both domain summary scores and player archetypes are inputs to development card compatibility.

**Resolution needed:** Specify the derivation method (simple average, weighted average, or something else) and clarify the relationship between domain scores and player archetype.

### M4 — "Trade leverage" listed as a reputation benefit; trades do not exist

**Files:** `2-season-structure.md` §2, `8-league-simulation.md` §1

`2-season-structure.md` §2 lists "trade leverage" as an effect of coaching reputation. `8-league-simulation.md` §1 explicitly states there are no trades. The term is an orphaned mechanic with no implementation target.

**Resolution needed:** Remove "trade leverage" from the reputation benefits in `2-season-structure.md`, or replace it with the correct free-agency equivalent.

### M5 — Situation card pool mid-season acquisition mechanism is undefined

**Files:** `5-card-systems.md` §2, `4-between-games.md` §2, `9-off-season.md` Chapter 6

`5-card-systems.md` §2 says the situation card pool is "rebuilt each season through between-game decisions (training sessions, film room work, scouting reports, narrative events)." `4-between-games.md` §2 lists training sessions and development card applications as between-game actions, but these appear player-development actions, not card acquisition. How new situation cards enter the pool mid-season (as distinct from the pre-season pool set in `9-off-season.md` Chapter 6) is never described.

**Resolution needed:** Clarify whether situation cards can be acquired mid-season through the between-game slot, or whether the pool is fixed at pre-season and only the 5-card selection changes each game.

### M8 — Development card timing: "between seasons" vs. "between games"

### M10 — Elite synergy situation cards: carry-over cap coverage undefined

**Files:** `10-relationships.md` §4, `5-card-systems.md` §2

`10-relationships.md` §4 states that elite player–player synergy "unlocks synergy-specific situation cards (analogous to relationship-tagged coach–player cards)." The 3-card carry-over cap is defined only for coach–player relationship cards. Whether synergy-specific cards count against the same cap, have their own cap, or are exempt is not stated. A roster with 2 elite synergies and 2 strong coach–player relationships could generate 4 eligible carry-over cards — exceeding the cap with no defined tie-breaking rule.

**Resolution needed:** State whether synergy situation cards are subject to the same 3-card cap, and define the tie-breaking rule when eligible cards exceed the cap.

### M11 — Carry-over cards are additive, but this is only explicit in one file

**Files:** `5-card-systems.md` §2, `0-design-notes.md` §3, `10-relationships.md` §3

Only `10-relationships.md` §3 explicitly states that carry-over cards are additive: "if a coach carries over 3 relationship cards, they start with those 3 *plus* the normal seasonal rebuild." `5-card-systems.md` §2 and `0-design-notes.md` §3 describe the cap without clarifying whether the carry-over cards replace slots within the ~15–20 pool or expand it beyond ~20.

**Resolution needed:** Add the additive clarification to `5-card-systems.md` §2 and `0-design-notes.md` §3.
